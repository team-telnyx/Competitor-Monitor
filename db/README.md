# Competitive Intelligence — database schema

Plain-SQL PostgreSQL migrations. This is the **contract** between the Python pipeline
(the producer — writes runs/pages/classifications) and the plg-dashboard `server/`
(the consumer — reads for the Competitive Intelligence page, writes operator feedback).
It is the plain-SQL port of `dashboard/backend/prisma/schema.prisma`; Prisma is not used
in the target repo.

## Layout
- `migrations/0001_init.sql` — Phase 1: read path + catalog (competitors, runs,
  run_competitors, pages, classifications, products, telnyx_offerings,
  offering_comparisons, snapshots). Ship this first.
- `migrations/0002_write_features.sql` — Phase 2: operator writes (feedback, guidance,
  removal_requests).

## Target database
`competitor_intel` in the **`pgbot-main-18`** cluster (Telnyx-managed Postgres /
CNPG), provisioned via the infra database-creation skill (PR `infra-data-pgbot#95`,
squad `solutions-eng-squad`). Credentials arrive as Vault path
`solutions-eng-squad/provided-creds/pgbot/pgbot-main-18/competitor_intel` and the K8s
secret `pg-role-competitor-intel`; consume the connection string as `DATABASE_URL`.

## Apply
```bash
psql "$DATABASE_URL" -f db/migrations/0001_init.sql
psql "$DATABASE_URL" -f db/migrations/0002_write_features.sql
```
Each migration is idempotent at the bookkeeping level (records itself in
`schema_migrations`) and wrapped in a transaction. Apply in order.

## Portability
Standard SQL + the standard **`pg` (node-postgres)** / **`psycopg`** drivers only — no
provider-specific drivers or features. Everything is reached via `DATABASE_URL`, so the
same schema/code runs against the bundled test Postgres and `pgbot-main-18` unchanged.

## Port notes (Prisma → Postgres)
- JSON-encoded `string[]` TEXT columns → native **`jsonb`** (`sitemap_urls`,
  `include_patterns`, `exclude_patterns`, `ignored_subdomains`, `products.aliases`,
  `snapshots.urls`).
- `@default(autoincrement())` → `bigint GENERATED BY DEFAULT AS IDENTITY`
  (BY DEFAULT, not ALWAYS, so the migration/data-load can set explicit ids).
- `@updatedAt` → the `set_updated_at()` BEFORE-UPDATE trigger (competitors, products,
  offering_comparisons).
- Documented status/enum strings are enforced with `CHECK` constraints. **Exception:**
  `classifications.category` is left free text — the taxonomy is mid-reconciliation
  (our 13-cat → PLG-67's 6 tags), so a CHECK would be premature.
- `DateTime` → `timestamptz`; `relevance_score` gets a `0–100` CHECK.
